# Navigation 의 공학적 설계 아이디어

본 문서는 자율주행 내비게이션 스택을 운영 가능한 형태로 만들기 위해 도입한 **컴퓨터공학적 핵심 아이디어** 만 정리한 것이다. 구현체(특정 라이브러리 이름) 가 아니라, 일반적인 알고리즘·시스템 패턴 수준에서만 기술한다.

---

## 1. 시간 척도 분리 — 전역 계획 vs 국소 제어

**문제.** 자율주행은 두 가지 본질적으로 다른 결정을 동시에 내려야 한다.

- *어디로 갈 것인가* — 환경 전체 위에서의 **장기 최적 경로** (정적 비용 위의 최적화)
- *지금 무엇을 할 것인가* — 동역학과 순간 장애물을 고려한 **단기 제어 명령** (실시간 회피·추종)

이 둘을 같은 알고리즘으로 풀면, 빠른 반응성과 전역 최적성을 동시에 만족시킬 수 없다.

**아이디어.** 두 결정을 **서로 다른 주기·비용 함수·알고리즘** 으로 분리한다.

- **Global planner** — 비용 그리드 위의 그래프 탐색 (A\* 계열). 호출 빈도 낮음(수 Hz), 입력은 *정적 환경 모델*.
- **Local planner** — 짧은 시간 지평의 trajectory rollout + 비용 함수(목표 추종 + 장애물 회피 + 동역학 제약) 평가. 호출 빈도 높음(수십 Hz), 입력은 *순간 센서*.

전역 계획기가 출력하는 경로는 국소 계획기에게 **참조 신호 (reference)**, 국소 계획기는 그 신호를 동역학적으로 가능한 속도 명령으로 변환한다.

**일반화.** 서로 다른 시간 척도의 문제를 한 알고리즘으로 풀지 않고, 계층화된 control hierarchy 로 분해 (hierarchical planning).

## 2. 비용 평면의 Layered 합성 (Layered Cost Representation)

**문제.** 주행이 회피해야 할 대상은 동질하지 않다.

- 변하지 않는 정적 환경 (벽, 가구)
- 실시간 관측되는 동적 장애물 (사람, 다른 로봇)
- 충돌 자체는 아니지만 가까이 가면 위험한 **안전 마진**

이를 단일 평면으로 한 번에 만들면, 한 종류 (예: 동적 관측) 의 갱신이 다른 종류(정적 지도) 를 침해한다.

**아이디어.** 비용 평면을 **독립적으로 갱신되는 layer 의 합성** 으로 표현한다.

- *Static layer* — SLAM 출력 지도, 거의 갱신 안 됨
- *Obstacle layer* — 실시간 센서 관측, 짧은 timeout 으로 휘발
- *Inflation layer* — 장애물 주변에 거리 함수 형태의 추가 비용을 부여, 안전 마진을 비용 함수에 인코딩

플래너가 보는 최종 비용은 이 layer 들의 합성. 각 layer 는 독립적인 갱신 정책을 가지면서도, 합성 결과는 단일 평면으로 추상화된다.

**일반화.** 서로 다른 출처·갱신 주기·신뢰도를 가진 정보를 **명시적으로 분리된 채널** 로 유지하다가 소비 시점에 합성. 관측의 의미적 분리 + 사용 시점의 통합.

## 3. 정책의 명시적 트리 표현 (Behavior Tree)

**문제.** 실세계 자율주행은 "경로를 따라간다" 만으로는 부족하다 — 경로 계획 실패, 국소 제어 실패, 장애물 갇힘 같은 **부분 실패** 가 빈번하고, 그때마다 정해진 복구 절차 (제자리 회전, 후진, 잠시 대기 후 재계획 등) 가 필요하다. 이 로직을 if/else 코드로 묻어두면 정책 변경이 곧 코드 수정이 된다.

**아이디어.** 주행 정책을 **선언적 트리** 로 표현한다.

- 트리 노드 = 단위 행동 (계획, 추종, 회복) 또는 제어 흐름 (sequence, fallback, parallel)
- 실패 시 fallback 분기로 자동 escalation (회전 → 후진 → 대기 → 재계획)
- 정책 변경은 트리 재구성 — 알고리즘 코드에는 손대지 않음

**일반화.** 정책 / 메커니즘의 분리(policy vs mechanism). 알고리즘은 메커니즘을 제공하고, 트리는 그것들의 조합 정책을 표현. 동일한 알고리즘 집합 위에서 서로 다른 정책을 구성할 수 있게 한다.

## 4. Frontier 기반 자율 탐사 — 미지 영역의 경계로 환원

**문제.** 사전 지도 없이 환경을 자율 탐사할 때, *어디로 다음에 가야 하는가* 라는 next-best-view 문제는 완전한 환경 모델이 없으면 정의되지 않는다.

**아이디어.** 미지 영역 탐사를 **"이미 안다(free) ↔ 모른다(unknown) 의 경계 (frontier)"** 검출 문제로 환원한다.

- 매핑 결과의 occupancy grid 에서 free 셀 ↔ unknown 셀의 경계를 찾는다
- 그 경계 위의 점들이 후보 목적지가 된다
- 정보 이득(미탐사 면적) + 이동 비용으로 후보를 정렬해 next-best goal 선택
- 유효 frontier 가 임계 미만이면 탐사 종료

**아이디어의 핵심.** 환경 모델이 *불완전한 채로* "다음에 봐야 할 곳" 을 정의 가능하게 만든 것 — 정보가 있는 영역과 없는 영역의 경계 자체가 가장 정보 이득이 큰 관측 후보라는 통찰. 결정에 필요한 정보를 *현재 알고 있는 것의 형태* 로부터 직접 끌어낸다.

**일반화.** Active perception / information gathering 에서 흔한 패턴 — 모델이 불완전할 때 *모델이 아는 것 자체가* 다음 행동을 정의한다.

---

## 5. 비동기 Goal-Oriented 액션 인터페이스

**문제.** 주행은 **장시간 실행되는 비결정적 작업** 이다 — 수십 초 단위로 걸리고, 도중에 취소되거나 진행 상황이 보고되어야 한다. 단순 RPC (요청–응답) 모델로는 cancel·feedback·중간 결과를 표현할 수 없다.

**아이디어.** 주행을 **비동기 action 인터페이스** 로 모델링한다.

- 클라이언트는 goal 을 send 하고 즉시 반환 (non-blocking)
- 서버는 진행 중에 feedback 을 push (현재 위치, 남은 거리)
- 클라이언트는 언제든 cancel 을 보낼 수 있고, 서버는 안전한 정지 절차로 전환
- 종료 시 result (성공/실패/취소 사유) 가 통지됨

이로써 long-running task 의 **lifecycle 자체** 가 인터페이스의 일부가 된다.

**일반화.** 시간이 걸리는 작업은 RPC 가 아니라 **promise / future / async generator** 같은 lifecycle-aware 인터페이스로 모델링해야 cancellation, progress, partial result 가 자연스럽게 표현된다.

---

## 정리

| # | 일반 문제 | 적용 패턴 |
|---|---|---|
| 1 | 다른 시간 척도의 결정을 한 알고리즘으로 풀 수 없음 | Hierarchical planning (전역 ↔ 국소) |
| 2 | 이질적 출처의 정보 통합 | Layered cost representation |
| 3 | 부분 실패 처리·정책 변경 | Behavior tree (policy/mechanism 분리) |
| 4 | 불완전한 모델 위에서의 다음 행동 결정 | Frontier (모델의 경계 = 정보 이득) |
| 5 | Long-running task 의 lifecycle 표현 | Async action 인터페이스 |

핵심 한 문장으로:
> **"단일 평면·단일 알고리즘·단일 호출로 풀 수 없는 결정을, 서로 다른 시간 척도·정보 종류·실패 모델로 분해해 합성한다."**
